# Сценарії використання (Use Cases) — Real-Time Chat API

## 1. Реєстрація нового користувача
* **Актор:** Гість (Неавторизований користувач)
* **Передумови:** Користувач не авторизований у системі.
* **Основний сценарій:**
  1. Гість відправляє POST-запит на `/auth/register` з даними: `username`, `email`, `password`.
  2. Система валідує дані (email має правильний формат, пароль > 6 символів).
  3. Система перевіряє, що користувача з таким email ще не існує.
  4. Система хешує пароль та зберігає нового користувача в БД.
  5. Система повертає статус `201 Created` і дані користувача (без пароля).
* **Альтернативні сценарії / Помилки:**
  * Невалідний email або короткий пароль: система повертає `400 Bad Request`.
  * Користувач з таким email вже існує: система повертає `409 Conflict`.

## 2. Авторизація (Вхід)
* **Актор:** Гість
* **Передумови:** Користувач має зареєстрований акаунт.
* **Основний сценарій:**
  1. Гість відправляє POST-запит на `/auth/login` з `email` та `password`.
  2. Система знаходить користувача за email.
  3. Система перевіряє збіг хешів паролів.
  4. Система створює сесію (або видає JWT токен у cookies).
  5. Система повертає `200 OK`.
* **Альтернативні сценарії / Помилки:**
  * Користувача не знайдено або пароль невірний: система повертає `401 Unauthorized`.

## 3. Отримання історії повідомлень чату
* **Актор:** Авторизований користувач
* **Передумови:** Користувач успішно авторизований (має валідну сесію/токен).
* **Основний сценарій:**
  1. Користувач відправляє GET-запит на `/api/messages/:conversationId`.
  2. Система перевіряє авторизацію.
  3. Система перевіряє, що користувач є учасником цієї бесіди.
  4. Система дістає повідомлення з БД (з пагінацією/лімітом).
  5. Система повертає `200 OK` та масив повідомлень.
* **Альтернативні сценарії / Помилки:**
  * Користувач не авторизований: повертається `401 Unauthorized`.
  * Бесіди не існує або користувач не є її учасником: `404 Not Found` або `403 Forbidden`.

## 4. Створення нового повідомлення (через REST)
* **Актор:** Авторизований користувач
* **Передумови:** Користувач авторизований.
* **Основний сценарій:**
  1. Користувач відправляє POST-запит на `/api/messages` із текстом повідомлення та ID отримувача.
  2. Система валідує текст (не може бути порожнім, довжина > 0).
  3. Система зберігає повідомлення в БД з поточним timestamp.
  4. Система повертає `201 Created`.
* **Альтернативні сценарії / Помилки:**
  * Текст повідомлення порожній: `400 Bad Request`.
